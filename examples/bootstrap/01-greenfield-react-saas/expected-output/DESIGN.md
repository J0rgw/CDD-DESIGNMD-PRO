# Greenfield SaaS Reference System

## Overview

This DESIGN.md was generated by CDD-DesignMD-Pro for a fresh
multi-tenant B2B SaaS application. The brand voice is calm,
information-dense, and engineering-flavoured — appropriate for a
collaboration tool that lives in users' workdays. The system supports
two visual configurations (light and dark mode) and one runtime axis
(per-tenant accent color, white-labeled).

The system has no domain-specific invariants. It does carry a formal
anti-pattern catalog because the team uses code review and the
product is intended to live for years.

## Colors

The palette is rooted in a sky-blue brand hue and an amber accent.
Neutrals come from the Tailwind slate ramp. Brand and amber primitives
sit at tier 1 and are not consumed directly by components; the
semantic layer (`primary`, `accent`, `surface`, `text`, etc.) is what
components reference.

- **Primary (`#0EA5E9`):** the brand sky blue. Used for high-emphasis
  actions and brand callouts.
- **Accent (`#F59E0B`):** the amber accent. Used for highlights and
  visual punctuation. Overridden per tenant at runtime.
- **Surface (`#F8FAFC`):** the baseline canvas.
- **Text (`#0F172A`):** body copy at maximum contrast.
- **Border (`#CBD5E1`):** structural separator.

Both primary and accent are bright enough that they need dark text
on top to clear WCAG-AA. The semantic tokens `text-on-primary` and
`text-on-accent` capture this and route through `gray-900`.

## Typography

The system uses **Inter** for narrative text and **JetBrains Mono**
for labels and inline data. The scale is conservative: a single
display level, one headline, two body sizes, and a single label.

- **`display-lg`** for hero treatments and empty states.
- **`headline-md`** for section titles in product UI.
- **`body-md`** is the default for prose; **`body-sm`** for metadata.
- **`label-md`** is monospaced and lightly tracked, for technical
  readouts.

## Layout

Layout follows a **12-column fluid grid** with a fixed max-width of
1280px on desktop. The spacing scale is 4px-based and named
(`xs`–`xxl`). `gutter` and `margin` are derived for grid contexts.

This system has no density axis: a single set of spacing values is
applied to every viewport. If the team later needs comfortable vs
compact density, that becomes a Phase 2 question on a future
BOOTSTRAP run.

## Elevation & Depth

The system uses **tonal depth**, not shadow. Hierarchy is communicated
by switching between `surface` and `surface-strong` and by border
emphasis. Shadow is reserved for ephemeral overlays.

## Shapes

Corners are softly rounded: `rounded.md` (8px) on interactive elements
and `rounded.lg` (12px) on containers. Status indicators and chips
use `rounded.full`.

## Components

Component tokens reference the semantic layer; never the primitive
layer. Variants for state (hover, pressed, disabled) live in sibling
component keys; only `button-primary-hover` is declared in the seed.

- **`button-primary`** is the high-emphasis action.
- **`button-secondary`** is the supporting action.
- **`card`** is the grouping container.
- **`badge-accent`** is the runtime-tinted chip used for tenant-
  branded highlights.
- **`caption`** is the tier for metadata text.
- **`divider`** is the 1px structural separator.

## Do's and Don'ts

- Do read tenant accent through the Theme provider; never reach for
  the CSS custom property directly.
- Don't introduce additional accent hues at the component layer —
  route through `colors.accent` so the branding axis remains the
  single source of truth.
- Do meet WCAG-AA contrast (4.5:1) on every text/surface pair.
- Don't theme primary or surface tokens at runtime; only the accent
  is tenant-driven in this system.
- Do use `colors.primary` only as a single high-emphasis action per
  screen; secondary actions go through `surface-strong`.
- Don't hardcode hex values or Tailwind arbitrary color classes; the
  anti-pattern catalog will flag both.

## Theming Axes

Two axes are declared:

- **`mode`** (`light`, `dark`) controls surface, text, and border
  tokens. Build-time; selected via a class on the document root.
- **`branding`** is runtime, sourced from `tenant.theme_primary`,
  and controls only `colors.accent`. No other tokens vary by tenant.

There is no `aesthetic` or `density` axis in this system; the
product has a single look and a single spacing scale. Both are
candidates for a future BOOTSTRAP extension if requirements change.

## Token Tiers

The token tree is partitioned into three tiers:

- **Primitive** holds raw values: the brand sky ramp, the amber
  accent, and the slate gray ramp. Primitives never reference
  other tokens.
- **Semantic** holds intent-bearing roles: `primary`, `accent`,
  `surface`, `text`, `text-on-primary`, etc. Semantics may
  reference primitives.
- **Component** holds component-scoped values that reference the
  semantic layer.

## Anti-Patterns

Five drift-catalog entries are declared:

- **`hardcoded-hex`** (warning) — literal hex outside the token
  files indicates colour drift.
- **`tailwind-arbitrary-color`** (error) — Tailwind arbitrary-value
  classes for color bypass the semantic layer.
- **`inline-style-color`** (info) — inline JSX color styles should
  be hoisted into stylesheets.
- **`tenant-accent-bypass`** (error) — reading the tenant accent
  CSS variable directly bypasses the runtime fallback contract.
- **`hardcoded-px-spacing`** (warning) — pixel literals in
  margin/padding/gap should reference `spacing.*`.

## Runtime Tokens

One runtime token is declared:

- **`colors.accent`** is sourced from `tenant.theme_primary`,
  scoped per-tenant, and falls back to `#F59E0B` (the amber
  accent) when no tenant override is present.

Components must read `colors.accent` through the Theme provider;
the `tenant-accent-bypass` anti-pattern flags any code that reads
the underlying CSS variable directly.
