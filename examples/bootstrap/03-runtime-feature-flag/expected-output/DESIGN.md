# Single-Theme SaaS with Experiment Overrides

## Overview

This DESIGN.md was generated by CDD-DesignMD-Pro for a single-theme
B2B SaaS dashboard. The product has no multi-tenancy and no
per-customer branding axis. It does, however, run visual A/B
experiments through a feature-flag platform: specific component
tokens may receive a runtime override during an experiment to
measure conversion against the default treatment.

The design system models this honestly. The CTA color is declared
as a semantic token (`colors.cta`) whose runtime value MAY come from
a feature flag, with a literal fallback that matches the brand
primary. The override is decoupled from any theming axis — there is
no "experiment axis" with enumerated values, because the override is
ad-hoc and target-specific, not a system-wide configuration.

## Colors

The palette is rooted in a saturated blue brand hue (`#2563EB`) with
an amber accent for highlights. Neutrals come from the slate ramp.

- **Primary (`#2563EB`):** the brand blue. Used for high-emphasis
  actions in the default treatment.
- **Accent (`#F59E0B`):** the amber accent. Used for highlights.
- **CTA (`{colors.primary}` by default):** the high-stakes CTA
  background. May be overridden at session scope by an active
  feature flag for visual A/B experiments.
- **Surface, text, border:** standard slate-based neutrals.

The blue primary is dark enough to clear WCAG-AA against light text
(`text-on-primary` and `text-on-cta` resolve to `gray-50`). The amber
accent pairs with dark text (`text-on-accent` resolves to `gray-900`)
because amber fails WCAG-AA against white.

## Typography

The system uses **Inter** across the board. The scale is conservative:
one display, one headline, two body sizes, and one label.

## Layout

A 12-column fluid grid with a 1280px max-width on desktop. The
spacing scale is 4px-based. There is no density axis.

## Components

Components reference the semantic layer. The `button-cta` component
exists as a separate slot from `button-primary` precisely because
its background may be overridden at runtime; keeping it isolated
prevents the experiment from accidentally bleeding into every
primary action.

## Theming Axes

One axis:

- **`mode`** (`light`, `dark`) controls surface, text, and border
  tokens. Build-time; selected via a class on the document root.

There is no `branding` axis (single-tenant) and no `aesthetic` or
`density` axis. Future BOOTSTRAP runs may add them if requirements
change.

## Runtime Tokens

One runtime token is declared, and it is **not** bound to any
theming axis:

- **`colors.cta`** is sourced from
  `feature-flag.experiment-cta-color-v2`, scoped to the user's
  session, and falls back to `#2563EB` (the brand primary literal)
  when no flag value is present.

This is the canonical example of a runtime override that the schema
explicitly permits: a runtime entry whose `path` is not the target
of any theming axis. The intent is captured in the design contract
rather than buried inside feature-flag glue code.

The `experiment-flag-bypass` anti-pattern flags any code that
patches the CTA background imperatively, bypassing the feature-flag
provider and the fallback contract.

## Do's and Don'ts

- Do read the experimental CTA color through the feature-flag
  provider; never reach for the underlying CSS variable or patch
  the DOM directly.
- Don't introduce additional CTA color variants at the component
  layer — route through `colors.cta` so the runtime contract
  remains the single source of truth for the experiment.
- Do meet WCAG-AA contrast on every text/background pair, including
  the CTA fallback and any value the feature flag may inject.
- Don't extend the runtime block to cover every component token
  "just in case"; runtime overrides decoupled from axes are
  intentionally narrow and per-token.
